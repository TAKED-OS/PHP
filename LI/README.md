## Инструкция по запуску проекта

### 1. Установить XAMPP

Для запуска проекта необходимо установить **XAMPP**.

После установки нужно открыть XAMPP Control Panel и запустить:

- Apache;
- MySQL.

### 2. Скопировать проект

Папку проекта нужно скопировать в директорию `htdocs`.

Пример пути:

```text
C:\xampp\htdocs\camp
```

Если папка проекта называется иначе, адрес в браузере тоже будет другим.

Например, если проект находится по пути:

```text
C:\xampp\htdocs\camp_project
```

то он будет открываться по адресу:

```text
http://localhost/camp_project/index.php
```

### 3. Создать базу данных

Откройте phpMyAdmin в браузере:

```text
http://localhost/phpmyadmin
```

Создайте новую базу данных, например:

```text
camp_db
```

### 4. Импортировать базу данных

В phpMyAdmin выберите созданную базу данных и импортируйте SQL-файл проекта:

```text
database.sql
```

Если файл `database.sql` отсутствует, таблицы можно создать вручную по структуре, указанной в разделе **“Структура базы данных”**.

### 5. Настроить подключение к базе данных

Откройте файл `db.php` и проверьте настройки подключения:

```php
<?php

$host = "localhost";
$user = "root";
$password = "";
$database = "camp_db";
$port = 3307;

$conn = new mysqli($host, $user, $password, $database, $port);

if ($conn->connect_error) {
    die("Ошибка подключения к базе данных: " . $conn->connect_error);
}

$conn->set_charset("utf8mb4");
```

Если MySQL работает на стандартном порту, нужно указать:

```php
$port = 3306;
```

Если база данных называется иначе, нужно изменить строку:

```php
$database = "camp_db";
```

на название вашей базы данных.

### 6. Запустить проект

После запуска Apache и MySQL откройте проект в браузере:

```text
http://localhost/camp/index.php
```

### 7. Создать администратора

После регистрации обычного пользователя можно назначить ему роль администратора через phpMyAdmin.

Для этого нужно открыть таблицу `users` и изменить поле `role`:

```text
user → admin
```

Также можно выполнить SQL-запрос:

```sql
UPDATE users
SET role = 'admin'
WHERE username = 'your_username';
```

После изменения роли нужно выйти из аккаунта и войти снова.

### 8. Проверка запуска

После успешного запуска должны быть доступны:

- главная страница;
- регистрация;
- вход в систему;
- список детей;
- просмотр комнат;
- административные функции для пользователя с ролью `admin`.

| Тип пользователя | Доступ |
|---|---|
| Гость | Главная страница, вход, регистрация |
| Обычный пользователь | Главная страница, список детей, просмотр комнат |
| Администратор | Полное управление детьми, комнатами и пользователями |

Проект должен включать:

- защищённые разделы сайта;
- безопасное хранение паролей с использованием хеширования;

- клиентскую и серверную валидацию данных;
- обработку ошибок;
- роль администратора;
- административные функции управления данными;
- модульную структуру приложения;
- документацию проекта.

Для выполнения работы была выбрана тема:

**Camp Room Allocation System — веб-приложение для автоматического распределения детей по комнатам в лагере.**

Приложение позволяет хранить данные о детях и комнатах, выполнять автоматическое распределение детей по комнатам, просматривать результаты распределения и вручную корректировать их при необходимости.

Основная задача приложения — упростить процесс расселения детей в лагере с учётом ограничений:

- мальчики и девочки не должны находиться в одной комнате;
- разница возраста детей в одной комнате не должна превышать 2 года;
- количество детей в комнате не должно превышать вместимость комнаты;
- при возможности учитываются пожелания детей жить вместе.

## Цель работы

Целью работы является разработка веб-приложения для автоматизации процесса распределения детей по комнатам в лагере.

Приложение должно позволять пользователям просматривать данные о детях и комнатах, а администратору — управлять этими данными, выполнять автоматическое распределение и вручную корректировать результат.

Для достижения цели были поставлены следующие задачи:

- разработать структуру базы данных для хранения пользователей, детей и комнат;
- реализовать регистрацию и вход пользователей;
- обеспечить безопасное хранение паролей с использованием хеширования;
- реализовать разграничение доступа по ролям `user` и `admin`;
- создать публичную главную страницу с динамическими данными из базы данных;
- реализовать форму добавления ребёнка с клиентской и серверной валидацией;
- реализовать поиск детей по нескольким критериям;
- реализовать CRUD-операции для управления детьми;
- реализовать управление комнатами;
- реализовать автоматическое распределение детей по комнатам;
- реализовать ручное перемещение детей между комнатами;
- добавить проверку ограничений при распределении и ручном перемещении;
- реализовать экспорт распределения в Excel;
- подготовить документацию проекта в формате README.

## Используемые технологии

Для разработки проекта были использованы следующие технологии:

## Сценарии взаимодействия пользователей с приложением

### Сценарий 1. Регистрация пользователя

1. Пользователь открывает главную страницу.
2. Переходит на страницу регистрации.
3. Вводит логин и пароль.
4. Подтверждает пароль.
5. Система проверяет данные.
6. Если данные корректны, создаётся новый пользователь с ролью `user`.
7. Пользователь может войти в систему.

### Сценарий 2. Вход пользователя

1. Пользователь открывает страницу входа.
2. Вводит логин и пароль.
3. Система проверяет логин и пароль.
4. Если данные корректны, создаётся сессия.
5. Пользователь получает доступ к защищённым страницам.

### Сценарий 3. Добавление ребёнка администратором

1. Администратор входит в систему.
2. Переходит на страницу добавления ребёнка.
3. Заполняет форму: имя, дату рождения, пол, группу, пожелания и примечания.
4. Система выполняет клиентскую и серверную проверку данных.
5. Возраст ребёнка автоматически рассчитывается по дате рождения.
6. Данные сохраняются в базу данных.

### Сценарий 4. Поиск ребёнка

1. Пользователь открывает список детей.
2. Вводит критерии поиска: имя, пол, группа, возраст от и возраст до.
3. Система выполняет поиск в базе данных.
4. Пользователь видит найденные записи.

### Сценарий 5. Автоматическое распределение детей

1. Администратор открывает страницу комнат.
2. Нажимает кнопку автоматического распределения.
3. Система очищает старое распределение.
4. Система анализирует список детей и комнат.
5. Дети распределяются по комнатам с учётом ограничений.
6. Результат сохраняется в базе данных.
7. Администратор видит обновлённое распределение.

### Сценарий 6. Ручная корректировка распределения

1. Администратор открывает страницу комнат.
2. Перетаскивает ребёнка из одной комнаты в другую.
3. JavaScript отправляет запрос на сервер.
4. Сервер проверяет существование ребёнка, существование комнаты, наличие свободного места, совпадение пола и допустимую разницу возраста.
5. Если условия выполнены, ребёнок перемещается.
6. Если условия нарушены, система показывает сообщение об ошибке.

### Сценарий 7. Доступ обычного пользователя

1. Обычный пользователь входит в систему.
2. Открывает список детей или комнаты.
3. Система позволяет только просматривать данные.
4. Кнопки редактирования, удаления и управления комнатами не отображаются.

### Сценарий 8. Создание администратора

1. Администратор входит в систему.
2. Переходит в раздел управления администраторами.
3. Вводит логин и пароль нового администратора.
4. Система проверяет корректность данных.
5. Пароль хешируется.
6. В таблицу `users` добавляется новый пользователь с ролью `admin`.

## Структура базы данных

### Таблица `users`

Таблица `users` хранит данные пользователей.

| Поле | Тип | Описание |
|---|---|---|
| id | INT | Уникальный идентификатор пользователя |
| username | VARCHAR | Логин пользователя |
| password_hash | VARCHAR | Хеш пароля |
| role | ENUM / VARCHAR | Роль пользователя: `user` или `admin` |
| created_at | TIMESTAMP | Дата создания пользователя |

Пример SQL:

```sql
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(100) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    role ENUM('user', 'admin') NOT NULL DEFAULT 'user',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### Таблица `children`

Таблица `children` хранит данные детей.

| Поле | Тип | Описание |
|---|---|---|
| id | INT | Уникальный идентификатор ребёнка |
| name | VARCHAR | Имя ребёнка |
| age | INT | Возраст ребёнка |
| gender | VARCHAR / ENUM | Пол ребёнка |
| group_name | VARCHAR | Группа или отряд |
| wishes | TEXT | Пожелания ребёнка |
| notes | TEXT | Примечания |
| birth_date | DATE | Дата рождения |
| room_id | INT | ID комнаты |
| created_at | TIMESTAMP | Дата добавления записи |

Пример SQL:

```sql
CREATE TABLE children (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(150) NOT NULL,
    age INT,
    gender VARCHAR(10) NOT NULL,
    group_name VARCHAR(100),
    wishes TEXT,
    notes TEXT,
    birth_date DATE NOT NULL,
    room_id INT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### Таблица `rooms`

Таблица `rooms` хранит данные комнат.

| Поле | Тип | Описание |
|---|---|---|
| id | INT | Уникальный идентификатор комнаты |
| room_number | VARCHAR | Номер комнаты |
| capacity | INT | Вместимость комнаты |

Пример SQL:

```sql
CREATE TABLE rooms (
    id INT AUTO_INCREMENT PRIMARY KEY,
    room_number VARCHAR(20) NOT NULL,
    capacity INT NOT NULL
);
```

## Безопасность приложения

В проекте реализованы следующие меры безопасности.

### Хеширование паролей

Пароли не хранятся в базе данных в открытом виде. Для хеширования используется функция:

```php
password_hash($password, PASSWORD_DEFAULT);
```

Для проверки пароля используется:

```php
password_verify($password, $user["password_hash"]);
```

### Сессии

После входа пользователя данные сохраняются в сессии:

```php
$_SESSION["user_id"] = $user["id"];
$_SESSION["username"] = $user["username"];
$_SESSION["role"] = $user["role"];
```

Сессии используются для проверки доступа к защищённым страницам.

### Разграничение доступа

Для защиты страниц используются функции:

```php
requireLogin();
requireAdmin();
```

`requireLogin()` разрешает доступ только авторизованным пользователям.

`requireAdmin()` разрешает доступ только администраторам.

### Защита от SQL-инъекций

Для запросов к базе данных используются подготовленные выражения:

```php
$stmt = $conn->prepare("SELECT id FROM users WHERE username = ?");
$stmt->bind_param("s", $username);
```

### Защита вывода от XSS

При выводе пользовательских данных используется:

```php
htmlspecialchars($value);
```

### Валидация данных

В формах используется клиентская и серверная валидация.

Пример клиентской проверки:

```html
<input type="text" required minlength="2" maxlength="150">
```

Пример серверной проверки:

```php
if ($name === "") {
    $errors[] = "Введите имя ребёнка.";
}
```

## Дополнительные аспекты

В проекте была реализована общая навигационная панель через отдельный файл `navbar.php`. Это уменьшает дублирование HTML-кода и упрощает поддержку интерфейса.

Также была создана отдельная страница `access_denied.php`, которая отображается пользователям без прав доступа. Это делает обработку ошибок доступа более понятной и аккуратной.

Для удобства разработки и демонстрации был реализован экспорт распределения в Excel.

Redis изначально рассматривался как дополнительный компонент для кеширования, однако финальная версия проекта использует MySQL как основную базу данных, чтобы обеспечить стабильную работу приложения.
